## Options Considered

### vote-first review gate

**Omega's view:**
Turn Alpha-Omega into a daily `git diff` checkpoint tool for a developer already using Claude Code and Codex: add `ao review --staged`, `ao review --branch main`, and `ao decide path/to/ADR.md`. Default to blind parallel memos plus deterministic Sigma, then escalate to critique only when recommendations diverge, confidence is low, or severity is high; this fits the current repo and aligns with the NeurIPS 2025 result that most gains come from ensemble diversity more than extra debate rounds (https://openreview.net/pdf/cf23542e165c5a7e0a313adf0f13edae69ee5138.pdf).
- Pros: Makes all 4 current advantages visible in one daily workflow: blind phase, Sigma, rich resolution states, and project memory; Adopts the strongest competitor pattern without copying their weakest part: checkpointed multi-model review from Octopus/Rubber-Duck style tools, but no LLM chairman; Lets you add ai-counsel-style convergence detection cleanly: stop after the blind phase when both brains already agree strongly; Keeps the product narrow and practical: pre-commit, pre-PR, pre-ADR, risky migration review; Produces artifacts a developer can act on immediately: top risks, dissent map, merge recommendation, missing tests, experiments
- Cons: Narrower than the current 'universal thinking tool' story; Needs git-aware context assembly and better file selection to avoid noisy prompts; If the quality delta over single-model review is small, users will not tolerate extra latency or cost; The current artifact format is still debate-centric and would need a more review-centric presentation

### memory-native decision graph

**Omega's view:**
Upgrade `.alpha-omega/` from passive transcript storage into active institutional memory: save a small JSON sidecar for every debate, tag files/subsystems/options, surface related past decisions automatically, and add commands like `ao recall auth`, `ao contradictions`, and `ao why`. This strengthens the most defensible long-term moat because competitors can copy debate flows faster than they can copy repo-specific decision history.
- Pros: Directly strengthens a unique current advantage instead of inventing a new one; Very compatible with stdlib-only Python: JSON sidecars, path tags, lexical retrieval, contradiction checks; Makes AO better every week for the same developer and repo; Creates concrete daily value even when no live debate is run, because prior decisions remain queryable
- Cons: Value compounds slowly, so it is weaker as a first growth wedge; Naive lexical retrieval may miss semantically similar decisions; Stale or low-quality past debates can pollute future context; Without a review checkpoint product, memory alone can feel like nice infrastructure around an unclear habit

### evidence connectors mcp-lite

**Omega's view:**
Adopt the useful part of ai-counsel's evidence-grounding model (https://github.com/blueman82/ai-counsel) without turning AO into a heavy MCP platform: build a small evidence-pack layer that can pull `git diff`, changed files, test output, CI logs, and optional `gh`/Jira/Linear context before the blind phase. This raises evidence quality and makes Sigma more trustworthy, while keeping AO's deterministic resolver as the arbiter.
- Pros: Would materially improve the quality of both blind memos and critiques; Makes AO useful beyond pure architecture discussion: audits, PR review, incident retros, migration planning; Lets the blind phase work on shared evidence rather than rhetoric; Can start with first-party local adapters only: git, rg, pytest, markdown files
- Cons: Highest complexity and failure surface: auth, missing CLIs, flaky commands, context explosion; Pushes against the current simplicity mandate more than the other options; Easy to drift into building a generic agent platform instead of a focused tool; Optional integrations are useful, but the first user problem can likely be solved without them

### zero-friction packaging

**Omega's view:**
Package AO for `pipx`/PyPI/`uvx`, add shell completions, and make install/init one command for a Claude Code + Codex user. This copies a real strength from successful tools: easy trial beats clever architecture nobody installs.
- Pros: Fastest way to reduce adoption friction; Fits the current minimal Python CLI structure well; Makes community growth and testing easier later; Good defensive move against better-marketed competitors
- Cons: Improves distribution, not differentiated value; Does not make the 4 strategic advantages more visible by itself; Risks polishing packaging before product habit is proven; A developer who already clones tools locally may not care much

## Assumptions

- A Claude Code + Codex power user will tolerate roughly 2 model calls and about 1-3 minutes of latency for higher-confidence review on important changes
- Claude and Codex continue to produce meaningfully different failure modes on real repo work, not just benchmark tasks
- Git context, changed files, and local test output are enough for the first wedge; full MCP integration can wait
- The deterministic Sigma resolver is more valuable as a trust feature than an LLM-chairman synthesis would be
- The first user is best served by a narrow daily workflow around diffs, risky changes, and ADRs, not by a broad generic debate surface

## Blind Spots Acknowledged

- **Omega:** I am probably overweighting the needs of a solo or small-team developer who lives in the terminal and underweighting team-facing distribution channels like GitHub Actions or PR comments. My competitor evidence is based on public docs and repo readmes, not hands-on use of every tool. The NeurIPS 'Debate or Vote' result is strongest for homogeneous settings, while AO's core bet is heterogeneous frontier models; that means I may be underestimating the value of a limited critique round after the blind phase. I also may be underrating how much packaging and onboarding matter before users can appreciate Sigma, blind enforcement, or memory.
