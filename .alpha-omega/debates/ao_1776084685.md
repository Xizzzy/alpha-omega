## Question

REVIEW GATE: Design 'ao review' command — a lightweight daily workflow that runs blind memos + Sigma on code changes without the full critique phase. Based on NeurIPS 2025 finding: ensemble diversity matters more than debate rounds. So for 80% of cases, blind memos + Sigma are enough. Full critique is escalation for disagreements. Concrete commands to design: (1) 'ao review' — review unstaged changes in cwd, (2) 'ao review --staged' — review staged changes, (3) 'ao review --branch main' — review all commits since diverging from main. Output should be practical: top risks, blind spots found, merge recommendation (safe/risky/needs-debate), dissent map if brains disagree. If brains agree strongly, skip critique phase (save time+tokens). If they disagree or confidence is low, auto-escalate to full debate. Design questions: How to build the diff context? Should it include file contents or just diffs? How to make output actionable vs academic? What threshold for auto-escalation to full debate? Should it save to .alpha-omega/ or be ephemeral? This is the primary daily-use command that makes AO a habit, not an event.

## Options Considered

### patch-first review gate

**Omega's view:**
Add a separate `ao review` path with a review-specific memo schema and deterministic `review_sigma`, instead of forcing code review into the current debate/option format. Build context from `git diff --unified=3 --no-ext-diff`, `--stat`, rename metadata, and full contents only for small or newly added changed files; default to ephemeral output, and auto-escalate to full debate only when the two blind memos disagree or look weak.
- Pros: Fits the repo direction already recorded in `.alpha-omega/decisions.md` toward a vote-first review gate; Matches the external evidence that majority voting and agent diversity often carry most of the gain, while extra rounds are not reliably worth the cost: https://openreview.net/forum?id=iUjGNJzrF1 and https://aclanthology.org/2025.findings-acl.606/; Keeps daily usage fast enough to become habit: 2 blind calls first, critique only on escalation; Practical output is easy to specify: `merge_recommendation`, top 5 risks, missing tests, blind spots, and dissent map; Patch-first context is cheap and grounded; conditional full-file inclusion covers new files and small edits without exploding prompt size; Ephemeral-by-default avoids polluting `.alpha-omega/` with low-signal daily reviews
- Cons: Needs a new review memo schema and a dedicated resolver; the current `sigma.py` is debate-shaped and not a clean fit; Patch-only context will miss some cross-file invariants and hidden call-site breakage on large refactors; Escalation thresholds can be too noisy if they are tuned from intuition instead of repo-level evals; Branch review over large divergence may still be too big for a lightweight flow without chunking or caps

### persistent review ledger

**Omega's view:**
Make `ao review` a first-class logged workflow: every run saves a structured artifact under `.alpha-omega/reviews/` with verdict, risks, confidence, and escalation outcome. This creates memory and trend data, but it spends complexity and storage on a daily path before proving that users want history for every review.
- Pros: Creates a searchable history of risky patterns, repeated blind spots, and escalation frequency; Would make future AO memory more grounded than only architecture debates; Easy to audit and compare blind-only reviews against escalated debates later
- Cons: Daily artifact spam is a real UX tax; Most reviews are low-value once the code is merged or fixed; permanent storage is often wasteful; Adds state-management work before the core review signal is proven; Makes the command feel heavier than a quick checkpoint

### full-file grounded review

**Omega's view:**
Review changed work by always sending full contents of every changed file, plus the diff, to both brains. This maximizes local grounding and catches some patch-level blind spots, but it undermines the whole point of a lightweight daily gate because token volume and latency scale badly on normal feature branches.
- Pros: Better at catching invariants that do not appear in the patch hunk; Simpler mental model than hybrid inclusion rules; Safer for new files, generated interfaces, and edits with wide semantic coupling
- Cons: Too expensive for the primary daily-use command; Large branches will blow prompt budgets quickly; More context is not always better; it can dilute attention away from the actual changed lines; Does not solve the need for escalation logic, so you still pay for critique later on hard cases

### always-debate review

**Omega's view:**
Keep `ao review` simple by reusing the existing full debate protocol every time: blind memos, critique, then Sigma. This minimizes new protocol work, but it contradicts the stated product goal for review as a habitual daily checkpoint and ignores the evidence that extra rounds often have diminishing or negative returns.
- Pros: Reuses almost all of the current AO machinery; Highest confidence on genuinely ambiguous or high-stakes changes; No need to invent a second resolver immediately
- Cons: Too slow and expensive for the 80 percent case; Makes AO an event again instead of a habit; Keeps the output debate-centric instead of review-centric; Wastes critique rounds on obvious agree/agree reviews

## Debate Exchange

### Alpha's critique of Omega

**Steelman:** Omega correctly identifies that code review is a fundamentally different shape than architecture debate—it must be fast enough to become a daily habit, grounded in actual code changes, and shouldn't pollute decision history with low-signal reviews. The patch-first approach with conditional escalation balances thoroughness with speed, and the research showing diminishing returns from extra debate rounds directly supports making blind+vote the primary path.

- [major] The review memo schema and escalation criteria are mentioned as needed but not actually designed. 'Disagree or look weak' is too vague—we need concrete thresholds: do we escalate when both say 'risky' but cite different top-3 risks? When confidence < 0.6? When one says safe and one says risky? Without this, the implementation will drift into ad-hoc heuristics.
- [moderate] Missing the workflow integration story entirely. When does a user invoke `ao review`—pre-commit hook, manual command before push, IDE integration? The habit-formation goal depends critically on friction and timing, but Omega doesn't address how this fits into the existing Claude Code session workflow or whether it should be automatic.
- [moderate] The 'conditional full-file inclusion for small or new files' logic is undefined. What's the byte/line threshold? How do you handle a new 200-line file vs a 5-line change in a 2000-line file? Without concrete rules, context assembly becomes a source of inconsistency and surprise.
- [minor] Test coverage checking is mentioned in 'missing tests' output but not designed. Does AO need to parse test files, understand naming conventions (test_*, *_test.py, *Spec.ts), or just flag 'no test files changed'? This is a high-value signal for code review that deserves explicit treatment.
- *Concession:* Omega is correct that ephemeral-by-default prevents pollution of `.alpha-omega/decisions.md` with low-signal daily reviews, and that the persistent ledger option spends complexity before proving users want review history.
- *Concession:* The research citations about diminishing returns from extra rounds (NeurIPS 2025, ACL 2025) are relevant and correctly applied. Multi-round critique has costs that aren't always justified, especially for routine reviews.
- *Concession:* Omega correctly identified that reusing the full debate protocol (always-debate option) contradicts the habit-formation goal and would make review feel like an event, not a checkpoint.

### Omega's critique of Alpha

**Steelman:** The strongest signal in the other memo is operational rather than substantive: the current protocol can fail before producing any usable critique when a model reaches `max_turns` or gets diverted into tool use. Charitably interpreted, that is an argument for keeping review flows short, bounded, and resilient to one-brain failure rather than depending on long multi-turn deliberation.

- [major] There is no actual design recommendation or option analysis here, so it cannot outweigh or refine the tradeoff analysis between patch-first review, persistent logging, full-file review, and always-debate review.
- [moderate] The failure mode is informative about protocol robustness, but it does not specifically argue for any alternative review architecture.
- [moderate] If anything, this failure weakens the case for always running full debate, because extra turns and tool interactions increase the chance of producing no review at all.
- *Concession:* I underweighted protocol-failure handling as a product requirement. A review path must degrade gracefully when one brain errors, times out, or returns malformed output.
- *Concession:* The error suggests that tool-use and turn-budget interactions are a real implementation hazard in AO, not just a hypothetical edge case.

## Assumptions

- A review-specific schema is acceptable; AO does not need to force every workflow through the current debate JSON shape
- Most daily reviews are on local git changes where `git diff`, `git diff --cached`, and `git merge-base <branch> HEAD` are available
- Patch-first context with conditional full-file inclusion is good enough for most review tasks
- Persisting only escalations or explicit `--save` runs is better than logging every review by default
- The main user tolerates 2 model calls and roughly 1-3 minutes for higher-confidence review, but not full debate latency on every diff

## Blind Spots Acknowledged

- **Omega:** I am likely biased toward the solo terminal-heavy power-user workflow already favored in this repo, which may underweight team-facing review modes like PR comments and CI integration. I may also be underestimating how often patch-only review misses semantic regressions in untouched files, especially in dynamic languages. The research I used is about multi-agent reasoning systems, not repo-local code review specifically, so I may be extrapolating too confidently from task benchmarks to engineering review workflow.
