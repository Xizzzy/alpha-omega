## Options Considered

### single-pass executor ledger

**Omega's view:**
Add `ao implement <session_id> --executor alpha|omega|winner` as one claimed execution attempt backed by `.alpha-omega/sessions/<id>.json`. AO builds a deterministic `implementation_brief`, runs the chosen brain once in implement mode, raises Alpha from `--max-turns 1` to `6`, keeps Omega on `--ephemeral`, uses a longer timeout like `900s`, and requires a structured completion report instead of another debate memo.
- Pros: Fits the current codebase: `BrainResult` already exists, debate artifacts already exist, and the missing piece is a richer session JSON plus an implement command.; The brief can stay deterministic: `goal`, `resolution`, `winning_option`, `winning_brain`, `winner_basis`, `sigma_reasoning`, `winning_thesis`, `winning_pros`, `constraints`, `dissent`, `open_questions`, `extra_files`, `validation_policy`, and required final report schema.; The executor prompt is clear and different from debate prompts: implement the resolved option, do not re-debate, edit files in the working tree, run focused checks, and end with a machine-readable report like `status`, `summary`, `files_changed`, `commands_ran`, `checks`, `blockers`, `next_steps`.; A simple lock file such as `.alpha-omega/sessions/<id>.lock.json` is enough to prevent double execution without building a broker.; Completion should show `git status --short`, `git diff --stat`, changed files, and validation results; that gives auditability without dumping a full patch.; Partial failure is manageable: record an `attempts[]` entry with `status=partial|failed`, preserve any working-tree changes, release the lock, and never auto-revert user code.
- Cons: Alpha in `--print` mode may still be brittle for larger implementations even with `6` turns.; Deriving a good brief from free-text debate artifacts may be sufficient for most sessions but weak for vague winning options.; Without resume semantics, a crash mid-run leaves recovery to a new attempt rather than continuing the same one.; `winning_brain` needs a deterministic rule when both brains converge on the same winning option.

### plan-then-apply implementation

**Omega's view:**
Split `ao implement` into two brain invocations: a short preflight that emits a structured patch plan, then an apply phase that edits code and validates it. This reduces ambiguity when the winning option is broad, but it adds another orchestration layer and more chances for drift between plan and execution.
- Pros: Makes `implementation_brief` richer by explicitly adding `target_files`, `acceptance_checks`, and ordered steps before any edits happen.; Better for risky changes because the preflight can fail fast on ambiguity, missing files, or unresolved assumptions.; Gives the CLI a natural place to show the plan before execution and to save a more useful audit trail.
- Cons: Costs roughly 2x the latency and tokens for every implementation attempt.; Creates a new failure mode where the plan is good but the apply phase ignores or mutates it.; Pushes AO closer to an agent workflow engine rather than a small deterministic orchestrator.; Still does not fully solve long-running recovery unless resume logic is added on top.

### resumable implementation jobs

**Omega's view:**
Turn implementation into a small job system with persistent run state, heartbeats, resumable attempts, and explicit `claimed`, `running`, `partial`, `failed`, `completed` states. This is the most robust path for long code changes, but it is close to the broker/daemon idea that the repo has already rejected as too heavy.
- Pros: Best handling of partial failure, interrupted CLIs, and long-running edits.; Makes lock ownership, stale claims, retry history, and resume semantics explicit instead of ad hoc.; Could support future features like `ao implement --resume` or delayed verification.
- Cons: Overengineered for the current stdlib CLI and local single-machine workflow.; Requires more state machinery than debate sessions currently have.; Harder to keep simple, testable, and symmetric across Claude and Codex wrappers.; Solves a problem size the project may not actually have yet.

## Debate Exchange

### Alpha's critique of Omega

**Steelman:** Omega proposes a pragmatic single-pass implementation command that builds a deterministic brief from debate artifacts, runs the chosen brain once with a structured completion requirement, and uses simple session-based locking to prevent concurrent execution—fitting cleanly into the existing architecture without adding workflow engine complexity.

- [major] The proposal creates unexplained asymmetry: Alpha gets `--max-turns 6` while Omega stays `--ephemeral`. If Omega wins the debate, why should it be handicapped during implementation? The whole point of AO is that neither brain is structurally weaker. This violates invariant #2 (both competent) unless there's evidence that Codex's ephemeral mode is actually sufficient for multi-file edits.
- [moderate] Deterministic brief assembly from free-text debate artifacts assumes the winning option is concrete enough to execute. But debates often produce high-level strategic options ('refactor for extensibility', 'optimize for performance') that need synthesis into specific file edits and tests. Without an LLM pass to translate debate resolution → implementation plan, the executor might guess wrong.
- [minor] Simple lock files without timeout or PID tracking leave stale locks after crashes. The proposal mentions `--force-claim` as an open question but doesn't specify default behavior when encountering an existing lock from a dead process. This creates manual cleanup friction.
- *Concession:* I failed to produce any analysis in the blind phase (error_max_turns), so Omega's structured breakdown of three options with scored tradeoffs is strictly more valuable than my non-output.
- *Concession:* The insight that implementation needs a fundamentally different prompt than debate ('implement the resolved option, do not re-debate, edit files, run checks, report') is correct and non-obvious. This is a distinct capability, not just debate continuation.
- *Concession:* Recording attempts in session JSON with status/summary/files_changed/validation provides auditability without brittle full-transcript logging. This is the right level of observability for a CLI tool.

### Omega's critique of Alpha

**Steelman:** The strongest reading of this result is that it provides concrete operational evidence, not speculation: the current Alpha path already fails under `max_turns=1`, so any implementation design that assumes Alpha can inspect, edit, and validate code is building on a known-bad default. In that sense, reliability of the executor wrapper is the first design constraint, and it should gate feature shape rather than be treated as a minor tuning detail.

- [major] It is not actually a competing design recommendation, so it does not resolve the main product question of what `ao implement` should look like.
- [moderate] The failure signal is real but narrow; by itself it does not justify a heavier orchestration model such as plan/apply or resumable jobs.
- [moderate] It leaves unresolved whether the problem is Alpha-specific, prompt-specific, or generally about long-running implementation, which matters for choosing the simplest fix.
- *Concession:* They surfaced hard evidence for my own stated blind spot: Alpha under the current wrapper is brittle enough that `--max-turns 1` is not a viable baseline for implementation.
- *Concession:* They also indirectly show that we should not treat implement-mode as 'just another debate call' with minor prompt edits.

## Assumptions

- A simple lock file plus attempt log is enough for now; full resume semantics are not required in v1.
- Both CLIs can edit files and run shell commands non-interactively in the target repo.
- Implementation should only be allowed for resolved sessions, practically `ADOPT` and `ADOPT_WITH_DISSENT`, unless a later `--force` policy is added.
- Showing diff summaries and validation output is more useful than showing full raw model transcripts after implementation.
- The session JSON can become the source of truth for post-debate execution without needing another LLM synthesis pass.

## Blind Spots Acknowledged

- **Omega:** I am probably underweighting how unreliable long tool-using runs can be under `claude --print`, even with a higher turn budget. I am also assuming that a deterministic brief assembled from debate artifacts is sufficient for implementation quality; if winning options are often high-level or rhetorical, that assumption breaks. Finally, I may be too willing to accept simple lock semantics because the repo values simplicity, while real user workflows may expose more git-state and recovery edge cases than the current design discussion has surfaced.
