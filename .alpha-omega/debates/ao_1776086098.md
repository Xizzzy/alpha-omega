## Question

MEMORY UPGRADE: Design 'ao recall' and 'ao contradictions' commands. Context: .alpha-omega/ already stores decisions.md (text log), debates/ (markdown transcripts), sessions/ (structured JSON), reviews/ (review JSON). The goal is to make this institutional memory ACTIVE, not passive. (1) 'ao recall <keyword>' — search past decisions, debates, and reviews by keyword. Show matching decisions with context. Should search across all artifact types (decisions.md entries, session JSONs, review JSONs, debate transcripts). Rank by relevance. (2) 'ao contradictions' — scan all decisions and find pairs that contradict each other. E.g., one debate decided 'use Python' and a later one decided 'rewrite in Node'. This is lexical/heuristic, not LLM — keep it stdlib-only and deterministic. How to detect contradictions without an LLM? Keyword overlap + opposite resolution? Same topic different winners? Design questions: What search index structure? How to rank results? What makes a 'contradiction'? Should ao recall show full debate or just summary? ## Options Considered

### lazy scan

**Omega's view:**
Do not persist a new index. On each `ao recall` or `ao contradictions`, walk `.alpha-omega/`, merge `decisions.md`, session JSON, review JSON, and debate markdown into in-memory canonical records, then rank matches with simple field weights and substring/phrase bonuses. This is the smallest change, avoids stale state, and is probably fast enough for low hundreds of artifacts.
- Pros: Lowest implementation risk and no index invalidation problem; Fits stdlib-only and current flat-file memory model; Easy to debug because results come directly from source artifacts; Good first step if the real corpus stays under a few hundred sessions
- Cons: Repeated parsing of markdown and JSON on every query wastes work; Ranking quality will be weaker unless the scan code becomes fairly elaborate; Contradiction detection becomes an expensive pairwise recomputation every run; Latency and code complexity will creep up as memory grows

### canonical index

**Omega's view:**
Build one deterministic `.alpha-omega/memory_index.json` with one canonical document per debate or review, not one result per file. Each doc merges all available sources for that item, stores normalized tokens and metadata, and is ranked by field-weighted lexical scoring for `ao recall`; the same index also supplies candidate pairs for `ao contradictions`, which classifies matches as `direct_opposite`, `exclusive_choice`, or `policy_reversal` using overlap thresholds plus a small antonym lexicon. Default `ao recall` should show compact summaries with snippets and source refs, not full debates; add `--full` only when needed.
- Pros: Best balance of speed, determinism, and simplicity for hundreds of debates; Avoids duplicate hits from `decisions.md`, `sessions/`, and `debates/` referring to the same session; Makes recall relevance better because title, winner, question, summary, and body can be weighted differently; Makes contradiction scanning cheaper by comparing only candidate documents that share rare topic terms; Provides a reusable foundation for future commands like `ao why` or automatic related-decision surfacing
- Cons: Adds index maintenance and schema-versioning work; Can become stale if artifacts are edited manually unless rebuild logic exists; Heuristic contradiction labels will still have false positives and false negatives; Needs careful normalization because the repo already mixes English and Russian text

### decision cards

**Omega's view:**
Introduce a new saved sidecar per resolved debate that extracts a compact `decision_card` with fields like `topic_terms`, `choice_terms`, `action_terms`, `polarity`, and referenced files. `ao recall` searches those cards first, and `ao contradictions` compares cards instead of raw text, which should improve contradiction precision if the extraction is reliable.
- Pros: Cleaner contradiction logic because comparisons happen on normalized fields instead of free text; Compact artifacts are cheap to scan and easy to inspect manually; Could become the backbone of a stronger memory system later
- Cons: Extraction heuristics are the hard part and current artifact text is too free-form to trust them fully; Likely to produce misleading certainty when winner names are vague or rhetorical; Higher design cost now for benefits the current corpus may not justify yet; Multilingual and mixed-format decisions make normalized card extraction brittle

## Assumptions

- For contradictions, only resolved debates should be first-class inputs; reviews are useful for recall but too patch-specific to drive policy contradictions
- Hundreds of debates is the target scale, so a single JSON index remains practical
- It is acceptable for `ao contradictions` to surface potential contradictions with reasons rather than claim semantic truth
- Session JSON and review JSON are the most reliable metadata sources; `decisions.md` and debate markdown are supporting sources
- The primary searchable unit should be a canonical memory item like `debate:ao_...` or `review:aor_...`, not raw files

## Blind Spots Acknowledged

- **Omega:** I am probably optimizing for the current repo shape and solo-developer CLI workflow more than for future team-scale memory or cross-repo usage. I may also be underestimating how messy free-text `winning_option` names are for contradiction detection; if options are often phrased abstractly, my heuristic categories will look cleaner on paper than they perform in practice. The biggest technical blind spot is language: purely lexical deterministic methods are much weaker once the memory mixes English, Russian, abbreviations, and renamed concepts.
