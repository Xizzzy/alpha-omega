## Options Considered

### python package + guided cli

**Omega's view:**
Keep Alpha-Omega in Python, but stop shipping it as a repo script. Publish it as a proper package with `pyproject.toml` and a console entry point `ao`, and make onboarding: `pipx install alpha-omega` (or `python -m pip install alpha-omega`), `ao doctor`, `ao init`, then `ao debate "..."`. This works because the tool is mostly subprocess orchestration, file I/O, and prompt assembly, which the current Python stdlib design already handles well.
- Pros: Best onboarding story without a rewrite: install once, get a real `ao` command, no absolute-path docs.; Fits the current codebase and the repo's explicit constraints: Python 3.9+, stdlib only, minimal abstraction.; Lets you add a crucial first-run command: `ao doctor` should check `claude` in PATH, `codex` in PATH, auth state, Python version, write access, and print exact remediation steps before the first debate fails.; A concrete user flow is easy to document: 1. install `alpha-omega`; 2. run `ao doctor`; 3. `cd` into project; 4. `ao init`; 5. run a canned first debate; 6. use `ao history` and `ao status` after that.; As of 2026-04-13, the 'Node alignment' argument is weaker than it sounds: Codex CLI's official docs describe it as Rust and installable via npm/Homebrew (`https://developers.openai.com/codex/cli`), and Claude Code's official docs now recommend a native installer, with npm kept for compatibility (`https://code.claude.com/docs/en/getting-started`).
- Cons: Still introduces Python as a dependency for users who otherwise only use packaged binaries.; You now own package publishing, versioning, release notes, and cross-platform install docs.; Windows onboarding can still be rough if users do not already have a sane Python and `pipx` setup.

### git clone + repo script

**Omega's view:**
Keep the current model: users clone the repo and run `python3 /path/to/ao.py`. This is the lowest-effort path for maintainers, but it feels like an internal tool rather than a product and keeps onboarding brittle.
- Pros: No packaging work and no release pipeline required right now.; Contributors can inspect and modify the tool immediately.; Matches the current repo layout with minimal change.
- Cons: Poor first impression: docs need absolute paths, manual updates, and implicit PATH assumptions.; No clean separation between 'user install' and 'developer checkout'.; Makes `ao init` feel less trustworthy because the tool itself is not installed like a real CLI.; Harder to recommend broadly in other repos, CI, or team docs.

### npm package + node rewrite

**Omega's view:**
Rewrite Alpha-Omega as a Node CLI and distribute it with npm. This can reduce ecosystem mismatch for users who already live in JavaScript tooling, but it is still a substantial rewrite for a tool whose core behavior is just orchestrating other executables and local files.
- Pros: Potentially cleaner fit for developers who already have Node installed for adjacent tools.; Makes future GitHub and JS-ecosystem integrations somewhat more natural.; Could simplify onboarding for users who do not already have Python.
- Cons: High rewrite cost with limited product gain: the hard dependencies are still `claude` and `codex`, their auth, and the debate protocol.; Undercuts the current 'stdlib-only, simple, Python 3.9+' discipline without proving user value first.; The premise is partly outdated as of 2026-04-13: Codex CLI is officially documented as Rust, not 'a Node tool', and Claude Code officially recommends native install over npm (`https://developers.openai.com/codex/cli`, `https://code.claude.com/docs/en/getting-started`).; Harder to reverse once the repo, packaging, and contributor expectations move to a new stack.

### mcp server first

**Omega's view:**
Make Alpha-Omega primarily an MCP server so Claude, Codex, and IDE clients can call it as a tool instead of invoking a local CLI. Strategically this is attractive, but it is a worse first onboarding experience than a small CLI because MCP adds protocol, transport, and client-configuration complexity before the product wedge is proven.
- Pros: Strong long-term integration story for agent-native workflows and IDE tool access.; Can eventually let users invoke AO from inside existing AI environments instead of switching terminals.; Matches the direction of the broader tool ecosystem; MCP's official transport model is JSON-RPC over HTTP/SSE or similar bidirectional channels (`https://modelcontextprotocol.io/specification/draft/basic/transports`).
- Cons: Does not make day-1 onboarding simpler; users still need to install and configure the server in each client.; Adds server lifecycle, transport, and debugging complexity to a project that is currently a single local CLI.; Harder to keep the 'just run `ao init` in any repo' promise.; Better as a second interface after the CLI is solid, not as the first product surface.

## Assumptions

- Maintainers are willing to own basic Python packaging and release hygiene.
- Most target users are developers who can install one extra runtime if the workflow is clean and the value is obvious.
- The core job of Alpha-Omega will remain orchestration of external CLIs, prompts, and local artifacts rather than deep SDK embedding.
- The first wedge is a local developer CLI for repo decisions, reviews, and architecture debates, not an agent-platform product.
- The repo wants to preserve the current constraints: Python 3.9+, stdlib only, simple codebase.
