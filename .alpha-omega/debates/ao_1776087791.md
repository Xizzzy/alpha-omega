## Options Considered

### doctor plus init

**Omega's view:**
Keep the existing command model, but make the defaults legible. `ao` with no args should stop dumping raw argparse help and instead print a short welcome plus the next command: `ao doctor`. `ao init` should create both `AGENTS.md` and a minimal `CLAUDE.md`, then run a non-blocking post-init check so the user leaves with one concrete next step instead of three missing-file surprises.
- Pros: Smallest code delta from the current CLI shape in `alpha_omega/cli.py`.; Answers the biggest current mismatch: docs say both brains read project memory, but bootstrap only creates `AGENTS.md`; this option fixes that without adding a new top-level command.; Concrete no-args text can be: `Alpha-Omega v0.2.0\nTwo independent AI brains. One resolved answer.\n\nFirst time here?\n  ao doctor\n  ao init\n  ao debate --mode audit \"Audit this project. What are the top 3 technical risks, the top 3 strengths, and the single highest-leverage next improvement?\"\n\nUse ao --help for full command list.`; Fits the Unix expectation that help stays explicit and side-effect free.
- Cons: Still makes first value a 3-command path: doctor, init, debate.; Puts too much burden on the user to mentally stitch together auth, bootstrap, and first-use flow.; Does not create a memorable one-liner equivalent to `husky init` or `eslint --init`.; Lower stickiness because the first debate still depends on the user reading and copying the right next command.

### setup command

**Omega's view:**
Add one canonical first-run path: `ao setup`. `ao` with no args should print a stateful welcome that points first-time users to `ao setup`, while `ao init` remains the lower-level primitive. `ao setup` should run doctor checks, create `.alpha-omega/`, generate both `AGENTS.md` and `CLAUDE.md`, and finish by printing one recommended sample debate command or running it with `--run-sample`.
- Pros: Best match to proven CLI onboarding patterns: guided init for setup-heavy tools (`husky init`, ESLint init) and explicit auth-first flows in Claude/Codex docs. Sources: https://typicode.github.io/husky/get-started.html https://eslint.org/docs/latest/use/getting-started https://developers.openai.com/codex/cli/reference https://developers.openai.com/codex/auth https://code.claude.com/docs/en/setup; Gives the repo a single answer to 'what do I run after pip install?': `ao setup`.; Concrete `ao` no-args output should be: `Alpha-Omega v0.2.0\nTwo independent AI brains. One resolved answer.\n\nStatus: not set up in this project yet\nNext step: ao setup\n\nWhat setup does:\n  1. Checks Claude + Codex\n  2. Creates .alpha-omega/, AGENTS.md, CLAUDE.md\n  3. Gives you a first debate that works on almost any repo\n\nAlready set up?\n  ao debate \"your question\"\n  ao review\n  ao history\n\nUse ao --help for the full command list.`; Concrete `ao setup` output should be: `Alpha-Omega Setup\n=================\nStep 1/3 Checking prerequisites\n  [ok] Python 3.11.9\n  [ok] Claude CLI\n  [warn] Codex auth missing\n         fix: codex login\n\nStep 2/3 Initializing project memory\n  [created] .alpha-omega/\n  [created] AGENTS.md\n  [created] CLAUDE.md\n\nStep 3/3 First useful run\nRun this now:\n  ao debate --mode audit \"Audit this project. What are the top 3 technical risks, the top 3 strengths, and the single highest-leverage next improvement?\"\n\nTip: add --run-sample to ao setup if you want AO to launch that debate for you.`; Concrete `ao init` behavior should be: create `CLAUDE.md` if missing, with a 5-section skeleton: `What this repo is`, `Architecture`, `Commands`, `Constraints`, `Alpha's mandate`; then print `Created CLAUDE.md (Alpha memory). Debates work now; replace the TODO lines later for better results.`; Answers the `ao init` question cleanly: yes, run doctor-style checks automatically, but as a summary/warning pass, not as a blocker. Missing CLIs should not prevent local bootstrap.
- Cons: Adds a new top-level command, so docs and mental model need updating.; If implemented carelessly, `setup` can become a grab-bag command that grows too much logic.; Need to keep `setup` idempotent and careful about not overwriting existing `CLAUDE.md` or `AGENTS.md`.; Sample debate quality will still vary on repos with almost no readable context.

### implicit first-run wizard

**Omega's view:**
Make `ao` itself detect first-run state and launch a minimal stdin wizard: check prerequisites, ask whether to create project memory, then offer to run a sample debate. This is the fastest path for a brand-new user, but it hides behavior behind a bare command and makes automation less predictable.
- Pros: Lowest friction for humans: the user can type `ao` and get led forward immediately.; Concrete prompt flow can be: `Alpha-Omega is not set up in this project yet.\nSet it up now? [Y/n]` then `Run a sample audit debate after setup? [Y/n]`.; Feels closest to the 'just start the CLI' posture users already know from model-native tools.
- Cons: More magic than this repo needs; hidden side effects on `ao` are harder to document and harder to trust.; Worse for scripts, screenshots, and support because `ao` means different things depending on state and TTY.; More branching logic than `setup command` for only modest UX gain.; Easy to overbuild into a pseudo-TUI, which conflicts with the repo's simplicity constraint.

## Assumptions

- A generic audit prompt is useful enough on most codebases to serve as the first 'aha' run.
- Creating `CLAUDE.md` with TODOs is better than leaving Alpha without project memory, as long as existing files are never overwritten silently.
- Keeping `ao init` as a lower-level primitive is valuable for advanced users and scripts even if `ao setup` becomes the recommended path.
- Most first-time users install Alpha-Omega specifically to try it on a real repo, not as a library API.
- Users will tolerate warnings during setup, but not dead ends where the tool only says something is missing.

## Blind Spots Acknowledged

- **Omega:** I may be overweighting discoverability and underweighting the Unix norm that bare commands should stay boring and composable. I also may be assuming too much from examples like Husky and ESLint: Alpha-Omega has slower, costlier first value than a linter or hook tool, so the real bottleneck may be debate latency rather than setup clarity. Finally, I may be too optimistic that a generic audit prompt produces a satisfying first result across non-code or sparse repos.
