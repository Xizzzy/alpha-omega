## Question

SYMMETRY & DUAL IMPLEMENTATION. Currently Alpha-Omega is asymmetric: only Claude Code can initiate debates, Omega (Codex) is a passive subprocess. Problem: (1) The tool should be equally usable from Codex CLI sessions — Omega should be able to initiate debates with Alpha. (2) After a debate resolves, implementation should not be limited to whoever initiated the session. If Alpha started the debate but Omega's approach won, Omega should be able to mechanically implement the solution (write code, edit files). Both brains are equal participants with equal agency. Questions to debate: How to make ao.py work symmetrically — callable from both Claude Code and Codex CLI? How to enable the 'loser' or 'winner' of a debate to implement the resolution regardless of who initiated? What changes to primitives.py and protocol.py are needed? Should there be a shared workspace or artifact handoff protocol? Consider: Codex CLI uses AGENTS.md, Claude Code uses CLAUDE.md — how do both discover and invoke ao.py?

## Options Considered

### docs-only symmetry

**Omega's view:**
Treat the current architecture as already shell-symmetric: both Claude Code and Codex CLI can invoke the same `python3 ao.py --project . debate ...` command, so the main change is to document that equally in `CLAUDE.md`, `AGENTS.md`, `README.md`, and `ao init` output. Post-debate implementation stays manual: whichever brain should code reads the saved transcript and acts in its native session.
- Pros: Almost no code risk; mostly docs and init-template changes; Preserves the current simple stdlib CLI design; Makes Codex initiation first-class without changing protocol semantics
- Cons: Does not solve the mechanical handoff problem; Keeps Omega as a debate subprocess with no structured route to implement the winner; Relies on humans to translate debate output into code tasks

### session ledger plus implement

**Omega's view:**
Keep one neutral orchestrator, but make it explicitly two-phase. In `primitives.py`, replace hardwired analysis wrappers with a generic `run_brain(brain, prompt, work_dir, mode)` result object so `mode="implement"` can use more turns and return richer metadata; in `protocol.py`, save the full structured session, derive `winning_brain`, build an `implementation_brief`, and support `ao implement <session_id> --executor alpha|omega|winner|other`. The shared workspace is just the same repo working tree; the handoff protocol is a machine-readable session file such as `.alpha-omega/sessions/<id>.json` plus an optional claim/lock file.
- Pros: Solves both asymmetries: either CLI can initiate, and either brain can implement; Preserves the blind phase because Alpha and Omega remain fresh subprocess children, not the outer session; Fits the current architecture: `ao.py` stays the single entry point and `protocol.py` remains the coordinator; Creates an auditable artifact for implementation, not just Markdown; Lets the loser implement the winner's design by prompt contract instead of human reinterpretation
- Cons: Requires widening the current thin decision record into a full session ledger; Needs explicit executor prompts to stop the implementing brain from re-debating instead of coding; Alpha's current `--max-turns 1` wrapper is too tight for real implementation work and must be changed carefully; Still assumes a local single-machine workflow rather than distributed workers

### patch handoff protocol

**Omega's view:**
Do not add execution automation inside AO. Instead, extend `protocol.py` and `artifacts.py` so every debate emits a normalized implementation packet: winning option, non-negotiable constraints, target files, acceptance checks, dissent notes, and maybe a skeleton patch plan in `.alpha-omega/sessions/<id>.json`; either Claude or Codex can then be instructed separately to implement that packet in its native session.
- Pros: Much simpler than building an execution phase into AO; Avoids nested agentic runs and the CLI-recursion edge cases they bring; Produces a clean, reusable handoff artifact for either brain
- Cons: Does not fully satisfy the requirement that AO mechanically enable implementation; Still depends on a human or external prompt to start coding; Creates room for drift between the debate resolution and the eventual implementation prompt

### broker daemon

**Omega's view:**
Promote AO from a CLI script to a persistent broker that stores sessions, queues jobs for Alpha and Omega, and lets either side claim post-debate implementation asynchronously. `protocol.py` becomes stateful job orchestration, and `primitives.py` becomes worker adapters rather than one-shot subprocess wrappers.
- Pros: Most explicit symmetry model; Can support delayed execution, retries, claims, and multi-step implementation; Would scale better if AO becomes a multi-user tool
- Cons: Overengineered for the current stdlib-only local CLI; Introduces service lifecycle, state, locking, and recovery complexity; Moves the project away from its current small-tool identity

## Assumptions

- A simple claim or lock file is sufficient to prevent accidental double-implementation for now
- Both CLIs can edit the same working tree safely enough for a local single-repo workflow
- Both `claude` and `codex` CLIs are callable from the same shell regardless of whether the outer interactive session is Claude Code or Codex CLI
- Saving full session state under `.alpha-omega/` is acceptable and does not violate the tool's simplicity goals
- The outer session should be treated as the operator, not as one of the scored debate participants

## Blind Spots Acknowledged

- **Omega:** I am reasoning mostly from the current code structure, not from repeated live tests of nested Claude/Codex execution, so I may be underestimating recursion and tool-budget failures in real sessions. I am also optimizing for the local single-machine CLI model already present in this repo; if the intended future is team-scale async usage, my recommendation is probably too conservative.
