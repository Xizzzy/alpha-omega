## Options Considered

### PR/ADR Copilot

**Omega's view:**
Сузить Alpha-Omega до очень конкретной работы: review важных PR, архитектурных решений и risky migrations. Люди уже пользуются AI для генерации кода; востребованным AO станет тогда, когда он будет ловить blind spots до merge и выпускать готовый артефакт: summary, risks, dissent map, test plan и ADR.
- Pros: Решает регулярную и дорогую задачу, а не демонстрирует абстрактную multi-agent идею; Хорошо отличается от Copilot/OpenHands/Aider: не делает вид, что заменяет разработчика, а повышает качество решений и review; Естественные интеграции очевидны: local git diff, GitHub Action, PR comment, ADR file, issue comment; Текущая архитектура уже близка: context builder, debate protocol, Sigma, artifact pack; Манифест здесь полезен как объяснение 'почему второй мозг', но не как основной landing page copy
- Cons: Это уже не 'универсальный thinking tool', а осознанно более узкий wedge; Нужен очень хороший UX вокруг diff/PR scope, retry, parse failures, latency и стоимости; Если результат не превосходит обычный single-model review по сигналу на минуту ожидания, привычка не возникнет

### Agent Builder Eval Harness

**Omega's view:**
Позиционировать Alpha-Omega как инструмент для AI-энтузиастов, которые строят свои agent workflows и хотят сравнивать topology, prompts, models и outcomes. Ценность здесь не только в debate, а в воспроизводимой проверке: single-agent vs dual-brain, стоимость, latency, acceptance criteria, quality deltas.
- Pros: Очень точно бьет в аудиторию AI-энтузиастов и builders, а не обычных end users; Позволяет доказать, где dual-brain реально лучше single-agent, а где нет; Создает естественный контент для сообщества: benchmarks, public debates, eval datasets, win-rate reports; Sigma и structured artifacts здесь становятся сильной стороной, а не внутренней механикой
- Cons: Риск стать meta-tool для узкой аудитории, а не инструментом ежедневной пользы; Онбординг сложнее: человеку нужно хотеть мерить качество, а не просто получить ответ; Без хороших baseline-режимов и repeatable tasks быстро превратится в 'интересно, но непонятно зачем'

### MCP Evidence Hub

**Omega's view:**
Сделать AO decision-layer поверх реальных источников данных: GitHub, docs, issue tracker, Notion/Linear/Jira, web search, локальный репозиторий. Тогда дебат идет не вокруг абстрактных мнений, а вокруг собранного evidence pack, и инструмент становится практичным для design, audit и planning.
- Pros: Сильно повышает полезность: debate grounded in evidence, а не в rhetoric; Хорошо совпадает с направлением экосистемы вокруг MCP и tool-connected agents; Расширяет рынок за пределы code review: product decisions, audits, vendor choice, incident analysis; Повышает trust, потому что артефакт ссылается на реальные источники
- Cons: Существенно увеличивает сложность интеграций, auth и failure modes; Есть риск начать строить platform before product-market fit; Для маленького stdlib-only инструмента это тяжелее всего поддерживать

### Manifesto-Led Community Brand

**Omega's view:**
Оставить продукт сравнительно легким, а рост строить через сильное интеллектуальное позиционирование: манифест, публичные debate transcripts, curated prompts, community challenges и каталог качественных кейсов. Это может дать внимание и ранних сторонников, но само по себе не делает инструмент привычным.
- Pros: Очень дешево и быстро исполнимо; У проекта появляется узнаваемый голос в crowded agent space; Манифест реально силен как long-form foundation text для комьюнити и контрибьюторов; Помогает объяснить принцип heterogeneity and critique лучше, чем сухая документация
- Cons: Высокий риск выглядеть как академическая демонстрация или арт-проект; Не создает loop ежедневного использования; Если манифест поставить на вход, прагматичный пользователь может не дойти до пользы; Без concrete workflows позиционирование быстро обгонит сам продукт

## Debate Exchange

### Alpha's critique of Omega

**Steelman:** Omega делает правильный стратегический выбор: сузить Alpha-Omega от абстрактного 'универсального thinking tool' до конкретного wedge — ловить blind spots в критических PR и архитектурных решениях до merge. Это решает реальную боль (дорогие production ошибки), естественно интегрируется в GitHub/CLI workflow, и текущая архитектура (protocol, Sigma, artifacts) уже готова для этого фокуса.

- [major] Omega не анализирует конкретный user journey от discovery до habit formation. 'Естественные интеграции очевидны' — это не план. Какой один wedge первый: git hook? GitHub Action? PR comment bot? VS Code extension? Без четкого первого touch point риск распылиться.
- [major] Все 4 опции обходят фундаментальный вопрос: как доказать early adopters, что Claude и Codex стабильно находят разные blind spots? Omega предполагает heterogeneity работает, но не предлагает способ это продемонстрировать до полноценного copilot.
- [moderate] Latency 2-6 минут признается в open questions как риск, но решения нет. Для real-time PR review это может быть deal-breaker. Асинхронный режим (запустил → уведомление) требует совершенно другого UX.
- [minor] Omega ставит 'Manifesto-Led Community Brand' последним по impact (0.56), но для OSS-проекта сильная философская рамка на старте может привлечь contributors раньше, чем polish copilot UX. Это недооценка community-led growth.
- *Concession:* 'Универсальный thinking tool' — это слишком абстрактно для первого release. Wedge strategy необходима.
- *Concession:* Success criterion Omega 'Если результат не превосходит single-model review по сигналу на минуту ожидания, привычка не возникнет' — это правильная метрика.
- *Concession:* Открытый вопрос 'debate vs review/decision/risk language' критичен для positioning. 'Debate' звучит академически, 'review with dissent map' — практично.
- *Concession:* Assumptions Omega более testable: целевая аудитория (AI-энтузиасты и разработчики), 2x cost acceptable for higher confidence, GitHub/CLI-first важнее UI.

### Omega's critique of Alpha

**Steelman:** Самый сильный доступный сигнал из их стороны не про позиционирование, а про операционную реальность: multi-step debate workflow сам по себе хрупок, и если второй мозг не успевает доставить полезный артефакт в ограниченный turn budget, это бьет по ценности продукта сильнее любой красивой стратегии. В этом смысле их 'мемо' неявно усиливает тезис, что надежность протокола, лимиты ходов и graceful degradation должны быть частью wedge, а не второстепенной инженерной задачей.

- [major] У другой стороны фактически нет содержательной продуктовой рекомендации, поэтому сравнить альтернативный wedge по существу невозможно.
- [moderate] Если трактовать этот результат как косвенный аргумент, он скорее говорит о проблеме исполнения протокола, а не о том, какой рынок или use case выбирать.
- [moderate] Отсутствие содержательного ответа само по себе подтверждает риск, который я уже отмечал: без очень хорошего UX вокруг retries, failures, latency и summarized output dual-brain experience ломается до того, как пользователь увидит ценность.
- *Concession:* Они не дали лучшую стратегию, но их сбой дал мне более сильное доказательство, чем было в моем memo: надежность протокола и деградация при лимитах ходов должны быть частью первого продукта, а не просто внутренней задачей.

## Assumptions

- GitHub/CLI-first workflow важнее красивого standalone UI на раннем этапе
- Манифест можно вынести в supporting essay, а front page и README переписать в painkiller language
- Наиболее ценный use case для heterogenous debate — review и decision support, а не raw code generation
- Основная ранняя аудитория Alpha-Omega — технические AI-энтузиасты и разработчики, а не массовый consumer market
- Пользователи готовы платить примерно в 2 model calls за higher-confidence review, если находки реально влияют на merge/no-merge decisions

## Blind Spots Acknowledged

- **Omega:** Я, вероятно, переоцениваю developer-tool wedge, потому что сам репозиторий и текущая архитектура естественно тянут в сторону GitHub/CLI/use cases. Это может занизить потенциал более широкой аудитории AI-энтузиастов, которым интереснее research, writing или product-strategy workflows. Я также вывожу спрос в основном из успеха adjacent tools и общей динамики agentic coding ecosystem, а не из прямых интервью пользователей Alpha-Omega. Наконец, я могу недооценивать силу манифеста как раннего community magnet: для OSS-проекта сильная философская рамка иногда важнее оптимального продуктового copy на первом этапе.
